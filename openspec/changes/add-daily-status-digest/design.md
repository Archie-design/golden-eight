## Context

管理員（3 位，全為 `is_admin = true` 且已綁定 LINE，同時本身也是學員）目前只能被動查詢後台。近期所有資料問題都是學員回報或偶然發現，後台從未主動示警。

既有素材比預期完整：
- `/api/cron/daily-reminder`（06:00）已算出 missing 名單與 `calcMonthStats` 的 `rate`/`remaining`/`targetScore`，但只 `console.log`（`route.ts:59-61`）。
- `line_user_id` 已收集（20 位活躍成員中 10 位已綁），管理員 3/3 全綁。
- LINE 官方帳號（Messaging API channel）已存在，僅缺 `LINE_CHANNEL_ACCESS_TOKEN` 與 push 實作。

關鍵限制：打卡邏輯日以中午 12:00 為界，06:00 時前一邏輯日尚未截止，「未打卡」名單含仍可補打者，不是事實。

> **⚠️ 假設一度被實測推翻、後由方案 Y 修復（見「實作發現」）**：上述第 2、3 點隱含「已收集的 `line_user_id` 可用於 Messaging API 推播」。最初的 Bot（舊 `@581etqxs`）與 Login 不同 Provider，該假設不成立；改在同一 Provider 下新建 Bot（`@341stkih`）後，現有 `line_user_id` 即可用，推播已實測送達。摘要產生與快照留存全程不受此曲折影響。

## Goals / Non-Goals

**Goals:**
- 管理員零習慣成本地每日收到可行動的學員狀態摘要（推播找人，而非人找看板）。
- 摘要每則都有新資訊，避免長期未打卡者洗版導致告警疲乏。
- 留下每日狀態事實軌跡，讓「這個人怎麼一步步掉下去」可回溯。
- 以管理員為收件人驗證推播管道與文案，作為未來推播給全體學員的前置。

**Non-Goals:**
- 不推播給非管理員學員（綁定率僅 10/20，且最常漏卡者多未綁定）——留待後續變更。
- 不做後台看板頁面（已評估：需要養成「每天開後台」習慣，風險是又一個沒人看的白工）。
- 不改動打卡、計分、月結、罰款邏輯。本變更為唯讀觀測 + 推播。
- 不處理 LINE 綁定率提升。

## Decisions

### 決策 1：排程 12:30，目標邏輯日取 `getCheckinDayTaipei() - 1`
- **理由**：12:00 為邏輯日邊界，之前的「未打卡」是猜測（成員仍可補打），之後才是事實。12:30 的名單確定、可行動，且學員清醒，管理員敲得動。30 分鐘 buffer 對齊月結 cron 排 13:00 的同一原則。
- **off-by-one 陷阱**：cron 於 12:30 執行時 `getCheckinDayTaipei()` 已翻為今日；要報的是剛截止的**前一日**。若直接用當日，會把才開始 30 分鐘的邏輯日判為全員漏卡。此與月結 cron 取上個月的 `m-2` 同類陷阱。
- **替代方案**：維持 06:00。否決——名單含未截止者，推播會誤擾；且該時段學員多在睡，管理員無法即時介入。

### 決策 2：「狀態為主 + 變化標記」而非純狀態或純變化
- **理由**：純狀態（報全部）會讓長期未打卡者（如某成員整月僅打 4 天）每日上榜，數週後管理員即停止閱讀——所有告警系統的典型死法。純變化則失去全局視野，可能漏掉緩慢惡化者。折衷為：完整呈現 + ⚡ 標記當日變化 + 長期缺席摺疊成單行。
- **核心機制**：`miss_streak` 由 N 累進至 N+1 **不產生事件**，只有跨越邊界（開始缺卡／回歸／跌破門檻／回到門檻／轉入長期缺席）才是事件。

### 決策 3：新增 `daily_status_snapshot` 表而非即時查詢
- **理由**：「今天變了什麼」必須與昨日狀態比對，需要昨日快照。且 `rate`/`passing` 事後重建不出來——補登、月結、工時修正都會改變歷史分數（本專案已因「重跑用現況回推歷史」踩過坑，`monthly_summary.level` 快照即為此而生）。快照 = 防污染。
- **哪些該落地**：`rate`/`passing`（事後不可重建）、`miss_streak`（供跨界判定）、`pushed_at`（冪等與重試）。
- **哪些其實不必落地但一併存**：`missed` 可由 `checkin_records` 推得，但與 `miss_streak` 同列寫入成本趨近於零，且使比對邏輯自足、不需回查打卡表。
- **替代方案**：不建表、每次即時算。否決——算不出「昨天是什麼狀態」，且無處記錄推播狀態。

### 決策 4：不排除管理員自身的學員狀態
- **理由**：3 位管理員同時是學員，均有達成率與罰款。排除會製造真實盲點。附帶效益：管理員先體驗「被這則訊息點名」的感受，為未來推播給學員校準語氣。

### 決策 5：`LONG_ABSENCE_DAYS = 7` 為常數
- **理由**：一週符合直覺。此為體感參數而非設計決策，定為常數以便依實際噪音調整。

### 決策 6：既有 `daily-reminder` cron 的處置
- 職責被本變更取代。建議**移除**其排程（`vercel.json`）與路由，避免兩支語意重疊的 cron 並存造成混淆。若欲保留其「06:00 前瞻提醒」語意，應另立變更明確定義其收件人與用途，而非留著 `console.log` 白工。
- 實作時應確認此決定，避免遺留死碼。

## Risks / Trade-offs

- [邏輯日 off-by-one 導致全員誤報漏卡] → spec 明訂 `D = getCheckinDayTaipei() - 1` 並以情境覆蓋；實作後以固定日期測試驗證。
- [首次啟用時無 `snapshot[D-1]`，全員被判為「新事件」造成首日爆量] → spec 明訂：缺前一日快照時不產生任何變化事件，僅寫快照並推無變化區塊的摘要。
- [新進豁免成員被誤判漏卡] → 依 `effective_start_date`（fallback `join_date`）排除，與工時分母、計分分母同源規則。
- [同日重跑重複推播] → `pushed_at` 標記 + 快照 upsert；推播失敗（未標記）則允許重跑補送。
- [LINE push 失敗使整支 cron 失敗、快照未寫入] → 先寫快照再推播；個別收件人失敗隔離、僅記 log。
- [`rate` 用 live 或 settled] → 月中無 settled，統一用當下 `calcMonthStats` 的 live 值；快照一旦寫入即凍結，不因月結而更新。
- [摘要日久仍可能疲乏] → `LONG_ABSENCE_DAYS` 可調；若仍吵，後續可再引入「僅在有 ⚡ 事件時推播」的模式。

## Migration Plan

1. 建立 `daily_status_snapshot` 表（migration；正式環境需於 Supabase SQL Editor 套用——PostgREST 無法執行 DDL）。
2. 設定 `LINE_CHANNEL_ACCESS_TOKEN`（Messaging API channel；與現有 Login channel 的 ID/SECRET 不同用途）。
3. 部署 cron 路由 + `vercel.json` 排程。
4. 首日僅寫快照、不報變化（設計已涵蓋），次日起摘要完整。
5. 回滾：移除排程即停止推播；快照表為唯讀觀測資料，保留無副作用。

## 實作發現：LINE userId 是 Provider-scoped，既有綁定對推播無效

**假設**：`members.line_user_id`（LINE Login 綁定時收集）可直接作為 Messaging API push 的收件人。

**實測結果**：不成立。三位管理員全數 `HTTP 404 Not found`。

證據鏈：
1. `GET /v2/bot/info` → Token 有效，Bot「星光黃金八套餐小幫手」(`@581etqxs`) 存在 → **排除 token 問題**（無效會回 401 而非 400）。
2. `GET /v2/bot/profile/{userId}` 對三位管理員全數 404。
3. 其中一位管理員**實際加了 Bot 好友後重測，仍為 404** → **排除「未加好友」**這個解釋。
4. 結論：LINE userId 是 **Provider-scoped**。Login channel（ID `2009794889`）與 Messaging channel 不在同一 Provider，故 Login 發出的 userId 在 Bot 視角下不存在。

**影響範圍**：僅投遞管道。快照產生、事件判定、摘要組版皆已驗證正常（邏輯測試 14/14、冪等測試通過）；`daily_status_snapshot` 每日仍忠實記錄狀態軌跡，其價值獨立於推播。管道接通後無需改動程式碼，摘要即自動送出。

### ✅ 已解決（方案 Y）

實際採用第三條路，比原先評估的 A/B 都省：**在 Login channel 所屬 Provider（「星光黃金八套餐小幫手」）下新建一個 Messaging Bot**「黃金八套餐小幫手」(`@341stkih`, `chatMode: bot`)。

- Login 與新 Bot 同 Provider → **現有 `line_user_id` 立即有效**，不需重綁、不需 webhook、不需新欄位。
- 實測：管理員柯啟鴻 `/v2/bot/profile` 可查到，且 push **實際送達手機**（截圖確認）。
- **程式碼零改動**：digest 收件人本就是 `line_user_id`，方案 Y 讓它回到「原本就正確」的狀態。
- 唯一前提：每位收件人須加**新** Bot `@341stkih` 好友（勿加舊的星光 `@581etqxs`，兩者名稱僅差「星光」二字，極易加錯）。

原評估的方案 A（搬 Login）／B（webhook）與獨立提案 `add-line-bot-webhook` 皆不再需要，應歸檔/作廢。

**教訓**：兩個同名 channel 分屬不同 Provider（Login 在「星光黃金八套餐」、舊 Messaging 在「星光西遊」），是 Provider-scoped 陷阱的根源。動工前應先目視確認 channel 的 Provider 歸屬，而非假設「有官方帳號 = userId 可用」。

### LINE 推播額度（成本天花板）

新 Bot 為免費方案：**200 則/月**（`/v2/bot/message/quota` 回 `{type: limited, value: 200}`）。計費按「訊息 × 收件人」，1 則發給 3 人 = 3 則。

- 現況（3 位管理員日報）：`3 × 30 = 90 則/月` ✅ 充裕。
- **未來擴大到全體學員（20 人日報）= `20 × 30 = 600 則/月`，超額 3 倍** ❌。屆時需擇一：只在有 ⚡ 變化時推、改週報、broadcast（不佔 push 額度但發給全體好友、無法個人化）、或升級付費方案。

## Open Questions

- 既有 `daily-reminder` cron 是移除或改寫（決策 6 建議移除，待實作時確認）。
- 是否需要回填歷史快照？目前傾向不需要——快照的價值在於「當下事實」，回填等同用現況回推歷史，正是本專案要避免的反模式。歷史趨勢若有需求，應另行設計。
- **門檻風險名單於月中全員誤報**：實測 7/16 觸發時 20/20 全數列入風險名單。因 `calcMonthStats` 分母為整月天數、分子僅累積至當日，月中 rate 天生偏低，故全員必然「未達標」。此名單月初 100% 誤報，違背本設計「每則都有訊號」的初衷，需另立變更調整判定（傾向以「依目前速度外推至月底」取代當下 rate 直接比門檻）。
