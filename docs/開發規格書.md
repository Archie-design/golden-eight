> 版本：1.3　　更新日期：2026-04-30
> 技術棧：Next.js 16 + Supabase + TypeScript

## 1. 系統概述

**黃金八套餐定課系統**是一套遊戲化每日打卡 / 習慣追蹤 App。成員每日完成 8 項任務並打卡記錄；每月累計達成率未達階梯門檻須繳罰款。系統供 21–50 位成員每日回報、追蹤個人進度，並提供管理員月結與罰款計算功能。

## 3. 業務規則

### 3.1 每日任務與計分

八項每日任務，各 1 分，最高 8 分／天：

| # | 任務 | 說明 |
|---|------|------|
| 1 | 早睡早起 | 12 前入睡、早上 7 前起床 |
| 2 | 破曉打拳 | 日出後建議開始時間（有連續天數統計） |
| 3 | 丹氣跑步 15 分鐘 | 雨天可室內 |
| 4 | 曬太陽 | 陰雨天不計分 |
| 5 | 工作 8 小時 | 依實際工時填寫（含假日）；月累計見 3.5 |
| 6 | 不吃肉 | 各素食級別皆可 |
| 7 | 寫觀心書 | 一階可寫覺察日記 |
| 8 | 淨心功法 | 睡前效果更好 |

**連續打拳加分（目前暫停）**：`punch_bonus` 欄位固定寫入 0，`total_score = base_score`。連續天數（`punch_streak`）仍正常記錄，供成就與月結統計使用。

### 3.2 打卡日邊界（12:00 規則）

打卡邏輯日以台北時間 **12:00** 為邊界：

- 12:00 前提交 → 歸入**前一日**打卡窗口（`getCheckinDayTaipei()` 回傳昨日）
- 12:00 後提交 → 歸入**當日**打卡窗口

範例：4/21 10:00 打卡記為 4/20；4/21 12:01 打卡記為 4/21。

已有記錄者拒絕重複提交（回 409）；不設獨立補報機制。

### 3.3 新成員起算計分日

函式 `computeEffectiveStartDate()`（`lib/api-helper.ts`）：

| 加入時間（台北） | effective_start_date | 最早可打卡時機 |
|-----------------|---------------------|----------------|
| < 12:00 | 加入日 +0（當天） | 當日 12:00 後 |
| ≥ 12:00 | 加入日 +1（隔天） | 隔天 12:00 後 |

**重要**：`/api/checkin/submit` 驗證「打卡邏輯日 < 起算日 → 409 拒絕」，確保分母不漏算。既有會員（`effective_start_date` 為 NULL）fallback 到 `join_date`。

### 3.4 戰士階梯與月結罰款

| 階梯 | 達成率門檻 | 未達罰款 |
|------|-----------|---------|
| 黃金戰士 | ≥ 80% | NT$200 |
| 白銀戰士 | ≥ 70% | NT$300 |
| 青銅戰士 | ≥ 60% | NT$400 |

每月 25 日後開放「選擇下月階梯」，月結時自動套用 `next_level` 並清空欄位。

### 3.5 月度結算流程

```
有效起始日    = max(月份第一天, effective_start_date)
有效滿分      = 有效天數 × 8
工時補扣      = ceil(max(0, 工作日×8 − 月累計工時) / 8) 分
調整後總分    = Σ total_score − 工時補扣
達成率        = 調整後總分 ÷ 有效滿分 × 100%
通過          = 達成率 ≥ 階梯門檻
罰款          = 未通過時依階梯扣款，通過則 0
破曉王        = 當月所有打卡日的 tasks[1] 全為 true
```

## 6. API 規格

### 6.1 共通規範

- 所有回應：`{ ok: true, ...data } | { ok: false, msg: string }`
- 認證：httpOnly Cookie `token`（JWT HS256）
- 時區：Asia/Taipei（UTC+8）
- CSRF 緩解：Origin ≠ Host 時回 403

### 6.2 /api/auth

| 方法 | 路徑 | 說明 | 參數 |
|------|------|------|------|
| POST | /login | 登入 | `{ name, phone, password? }` |
| POST | /register | 註冊 | `{ name, phone, joinDate?, level }` |
| GET | /me | 目前用戶（自動續簽 7d cookie） | — |
| POST | /logout | 登出（清 cookie） | — |
| POST | /next-level | 設定下月階梯（25 日後） | `{ level }` |
| POST | /set-password | 設定 / 修改密碼 | `{ password, currentPassword? }` |
| GET | /line | 產生 LINE OAuth URL（已登入） | — |
| DELETE | /line | 解除 LINE 綁定 | — |
| GET | /line/login | 未登入者 LINE 登入起點 | — |
| GET | /line/callback | LINE OAuth 回呼 | `?code&state` |

**登入細節**：
- 識別：姓名 + 完整 10 位手機（09 開頭）+ 密碼（已設者必填）
- 失敗鎖定：同一帳號連續失敗 5 次 → 鎖 15 分鐘（`failed_attempts` / `locked_until`）
- 舊帳號（只有 `phone_last3`）：首次以完整號碼登入時自動遷移
- 首次登入未設密碼：後端允許；`(main)/layout.tsx` 偵測後強制導向 `/setup-password`

### 6.3 /api/checkin

| 方法 | 路徑 | 說明 | 參數 |
|------|------|------|------|
| POST | /submit | 提交打卡 | `{ tasks: boolean[8], note?, work_hours? }` |
| PATCH | /submit | 修改當日打卡（誤觸回溯） | 同上 |
| GET | /today | 今日狀態 | — |

**POST 回應**：`{ ok, totalScore, baseScore, punchStreak, newAchievements[] }`

**PATCH 行為**：
- 僅允許修改當日邏輯日（`getCheckinDayTaipei()`）；無既有記錄 → 404；早於起算日 → 409
- 重算 base_score / total_score / punch_streak
- 成就對帳（`reconcileAchievementsAfterEdit`）：新增確定達成者；撤銷保守規則（只撤 DAILY_PERFECT / DAILY_PERFECT_BONUS / PERFECT_10/30，streak 與 CHECKIN_* 視歷史里程碑保留）
- 寫 `checkin_edit_logs` 稽核快照
- 回應：`{ ok, totalScore, baseScore, punchStreak, achievementsAdded, achievementsRemoved }`

**GET /today 回應**：`{ today, calendarDay, sunrise, punchStart, punchStreak, monthRate, todayRecord }`
- `today` = 打卡邏輯日；`calendarDay` = 實際日曆日；`punchStart` = 日出 +12 分

## 7. 業務邏輯函式（lib/scoring.ts）

```
calcBaseScore(tasks[8])
  → 0–8（完成任務數）

calcMonthStats(member, records[], refDate)
  → { maxScore, totalScore, rate, targetScore, remaining, passing }

calcWorkHoursDeduction(totalWorkHours, workingDays)
  → 補扣分數（每少 8 小時扣 1 分）

calcPenalty(level, passing)
  → 0 | 200 | 300 | 400
```

## 8. 成就系統規格（45 項）

單一事實來源：`lib/constants.ts`（`ACHIEVEMENT_LIST`）。

### 8.1 類型 A — 任務連續類（8 任務 × 4 里程碑 = 32 項）

每項任務均有 3 / 7 / 30 / 100 天連續成就，代碼格式 `T{n}_STREAK_{days}`：

| 任務 | 3 天 | 7 天 | 30 天 | 100 天 |
|------|------|------|-------|--------|
| T1 早睡早起 | 早鳥初心 | 早鳥習慣 | 早鳥達人 | 早鳥百日 |
| T2 破曉打拳 | 破曉初煉 | 破曉星火 | 破曉月將 | 破曉百日俠 |
| T3 丹氣跑步 | 跑步初動 | 跑步習慣 | 跑步達人 | 百日跑者 |
| T4 曬太陽 | 初曬太陽 | 陽光習慣 | 陽光達人 | 百日暖陽 |
| T5 工作8小時 | 勤奮初心 | 勤奮習慣 | 職人達人 | 職人百日 |
| T6 不吃肉 | 素心初願 | 素心習慣 | 素食達人 | 素食百日 |
| T7 寫觀心書 | 觀心初啟 | 觀心習慣 | 觀心達人 | 觀心百日 |
| T8 淨心功法 | 淨心初動 | 淨心習慣 | 淨心達人 | 淨心百日 |

連續天數判斷使用「打卡邏輯日」連續性（`calcTaskStreak`），105 日滑動窗口足以覆蓋最長 100 天里程碑。

### 8.2 類型 B — 單日特殊類（3 項）

| 代碼 | 名稱 | 觸發條件 |
|------|------|---------|
| FIRST_CHECKIN | 萬里起行 | 第一次打卡 |
| DAILY_PERFECT | 大滿貫 | 單日 base_score = 8 |
| DAILY_PERFECT_BONUS | 金色大滿貫 | total_score ≥ 8.5 ⚠️ 見 8.6 |

### 8.3 類型 C — 月度類（5 項，月結時觸發）

| 代碼 | 名稱 | 條件 |
|------|------|------|
| MONTH_PASS | 初次通關 | 首次月結通過 |
| MONTH_GOLD | 黃金通關 | 以黃金戰士通過月結 |
| MONTH_PERFECT | 完美月 | 月達成率 ≥ 100% |
| MONTH_STREAK_3 | 三月連勝 | 累計通關月份數 ≥ 3 |
| MONTH_STREAK_6 | 半年英雄 | 累計通關月份數 ≥ 6 |

> 破曉王以 `monthly_summary.is_dawn_king` 旗標記錄，不佔 achievements 表。

### 8.4 類型 D — 累積里程碑（5 項）

| 代碼 | 名稱 | 條件 |
|------|------|------|
| CHECKIN_30 | 打卡 30 天 | 累計打卡 ≥ 30 天 |
| CHECKIN_100 | 打卡百日 | 累計打卡 ≥ 100 天 |
| CHECKIN_365 | 打卡一年 | 累計打卡 ≥ 365 天 |
| PERFECT_10 | 大滿貫 x10 | 累計大滿貫 ≥ 10 次 |
| PERFECT_30 | 大滿貫 x30 | 累計大滿貫 ≥ 30 次 |

### 8.5 已知問題

**MONTH_STREAK 歷史上從未觸發（已於 v1.3 修復）**

原始實作以 `alreadyUnlocked.filter(c => 'MONTH_PASS').length` 計算通關月份數，但 `MONTH_PASS` 受 UNIQUE 約束每人至多一筆，計數恆為 0 或 1，永遠達不到 ≥ 3 的門檻。

v1.3 修正：`calcMonthlyAchievements` 新增 `passingCount` 參數，由 settlement route 從 `monthly_summary` 資料表統計後傳入，確保計數正確。**既有應獲獎但未發放的成員，需執行 `POST /api/admin/backfill-achievements` 回填。**

### 8.6 功能凍結項目

**DAILY_PERFECT_BONUS（金色大滿貫）— 暫停觸發**

觸發條件 `total_score ≥ 8.5` 依賴連續打拳加分（`punch_bonus = 0.5`），而此加分功能目前暫停（`punch_bonus` 固定寫入 0）。此成就定義保留於 `constants.ts`，不回填、不刪除，待連續打拳加分功能重新啟用後同步解封。

### 8.7 未來擴充方向

| 建議項目 | 說明 |
|---------|------|
| CHECKIN_200 / CHECKIN_500 | 填補 100→365 天的激勵空窗 |
| MONTH_FULL_CHECKIN | 某月每天皆有打卡記錄（月曆天數 = 打卡筆數） |
| DAWN_KING | 首次當選 / 累計三次破曉王的成就 |

## 9. 認證流程

```
識別：姓名 + 完整手機號碼（10 位）+ 密碼（已設者必填）
密碼：scrypt hash（lib/password.ts）；NULL = 未設定，登入後強制導向 /setup-password
失敗鎖定：連續失敗 5 次 → 鎖定 15 分鐘（per-IP rate limit 同時擋 DoS）
舊帳號遷移：首次以完整號碼登入時自動寫入 phone_hash
JWT：HS256，7d TTL，payload { sub, isAdmin, tv }，httpOnly Cookie
Sliding renewal：/api/auth/me 每次呼叫重新簽發 7d cookie
授權驗證：getCurrentMember() 驗 token + 比對 token_version + 查 members 表
Admin 驗證：requireAdmin() 每次查 DB 確認 is_admin（不信任 JWT payload）
LINE 登入：/line/login → LINE OAuth → /callback → 以 line_user_id 找帳號
LINE 綁定：已登入狀態下走 /line → OAuth → postMessage 回 dashboard
```

## 10. 安全規格

| 控制項 | 實作 |
|--------|------|
| Input Validation | 全 API Zod schema（`lib/validation.ts`） |
| JWT_SECRET | 強制 ≥ 32 字元 env，無 fallback |
| Admin 授權 | `requireAdmin()` 每次查 DB，不信任 JWT |
| 電話號碼 PII | HMAC-SHA256 `phone_hash`（PHONE_PEPPER） |
| JWT 撤銷 | `token_version` 欄位 + `tv` claim |
| Cookie | 生產環境 `secure: true`（`lib/cookie-options.ts`） |
| Rate Limiting | login 10/min、register 5/10min、checkin 20/min per-IP in-memory |
| 帳號鎖定 | 5 次失敗 → 15 分鐘鎖定 |
| CSRF 緩解 | Origin ≠ Host → 403（`checkOrigin()` in api-helper.ts） |
| RLS | 所有 Supabase 表啟用 RLS，service_role 繞過 |
| 原子操作 | `replace_schedule_template()` / `remove_tag_from_templates()` RPC |
| 成就計算 | 聚合 counter + 105 日滑動窗口（無全量掃描） |
| CSV 安全 | `lib/csv.ts` 跳脫公式注入字元（`= / + / - / @ / \t / \r`） |

## 15. 驗收測試規格

### 核心打卡流程
- 登入 / 登出正常，JWT Cookie 有效；首次登入自動導向 /setup-password
- 今日打卡提交，`checkin_records` 寫入正確
- 打卡日邊界：11:59 打卡記為前日；12:01 打卡記為當日
- 連打 3 天 `punch_streak` 正確累加；FIRST_CHECKIN / DAILY_PERFECT 成就正確觸發
- 「修改今日」：PATCH 重算後前端正確顯示新解鎖 / 撤銷成就

### 月結流程
- 管理員執行月結 → `monthly_summary` 寫入正確（含工時補扣）
- 達成率低於門檻 → penalty 正確；月結選出 is_dawn_king
- 三月連勝：第 3 次月結通過時解鎖 MONTH_STREAK_3（確認 v1.3 bug 修復生效）

### 儀表板月份回查
- 點擊左箭頭切換至上個月，資料正確切換
- 右箭頭在當月時為停用狀態
- 歷史月份連續打拳卡顯示「最終連續天數」

### 成就系統
- 打卡 30 / 100 天正確解鎖 CHECKIN_30 / CHECKIN_100
- T2_STREAK_7（破曉打拳連續 7 天）正確觸發
- DAILY_PERFECT_BONUS 不可解鎖（punch_bonus 固定 0）

### 行程模板
- 標籤庫 → 區段拖拉、區段間移動、區段內排序、拖回移除
- 儲存後重新載入資料一致；公開模板在群組行程可見

### 管理後台
- 非管理員無法存取 /admin 及 /api/admin/*
- 月份選擇器可查歷史罰款；月結後可匯出 CSV（Excel 開啟中文不亂碼）
- 成就統計 Tab：稀有度排序 + 成員解鎖進度

### 安全
- CSRF：Origin ≠ Host → 403
- Rate limit：同 IP 超過限額 → 429 含 Retry-After
- 停用帳號嘗試登入 → 401
- 非管理員呼叫 admin API → 403
