## 第二輪資安補丁與密碼功能（2026-04-20）

### B1（HIGH）— JWT TTL 與 cookie maxAge 不符
**檔案**：`lib/auth.ts`
**問題**：`createToken` 的 TTL 硬編為 `'30d'`，但 cookie `maxAge` 為 7 天。cookie 過期後 JWT 仍有效達 30 天，若 token 被從日誌或快取中取得仍可被重放。
**修復**：移除 `const TTL = '30d'`，改為 `import { AUTH_TOKEN_MAX_AGE } from '@/lib/cookie-options'` 並以 `Math.floor(Date.now() / 1000) + AUTH_TOKEN_MAX_AGE` 設定 `setExpirationTime`，確保 JWT `exp` 與 cookie 存活時間一致（7 天）。

### B2（MEDIUM）— LINE callback bind 不驗 token_version
**檔案**：`app/api/auth/line/callback/route.ts`
**問題**：bind 段落呼叫 `getTokenPayload()`（僅驗簽章，不查 DB），跳過了 `token_version` 比對。若成員的所有 JWT 已被撤銷（`token_version` 已遞增），仍可使用舊 token 綁定 LINE 帳號。
**修復**：bind 段落改用 `getCurrentMember()`，其內部包含 DB 側 `token_version` 比對，確保撤銷後的 token 無法執行 LINE 綁定。

### B3（LOW）— lineContext cookie 未白名單化
**檔案**：`app/api/auth/line/callback/route.ts`
**問題**：`lineContext` 直接取 cookie 原始值，攻擊者若能控制該 cookie 可注入任意字串。雖下游只對 `=== 'login'` 分支，但原始值仍可能外洩至 redirect URL。
**修復**：改為 `const lineContext = raw === 'login' ? 'login' : 'bind'`，明確白名單，任何非 `'login'` 的值皆視為 `'bind'`。

### B4（LOW）— Logout 未清除 state cookies
**檔案**：`app/api/auth/logout/route.ts`
**問題**：登出僅清除 `token` cookie，`line_state` 與 `line_context` 殘留。若下一個使用者在同一裝置發起 LINE OAuth，可能重用舊的 state。
**修復**：登出時同步清除 `line_state` 與 `line_context`。

### FEAT — 密碼機制 + 首次登入強制設定
**背景**：社群成員彼此熟識，手機號相對容易取得，名稱+手機的驗證強度不足。
**設計原則**：
- **漸進遷移**：既有帳號 `password_hash IS NULL`，首次仍可用名稱+手機登入，但 layout 強制 redirect 到 `/setup-password`。
- **不可逆雜湊**：`crypto.scrypt`（Node.js 內建，無新套件）+ 16-byte 隨機 salt + `timingSafeEqual` 常數時間比對，管理員查 DB 只看到不可逆 hash。
- **首次設定**：不需現有密碼；後續修改需先驗現有密碼。

**新增/修改檔案**：

## 第三輪資安補丁（2026-04-22）

### C-1（CRITICAL）— `phone_last3` 3 碼暴力破解漏洞
**檔案**：`app/api/auth/login/route.ts`
**問題**：
登入比對存在雙重路徑——優先比 `phone_hash`，若 hash 為 null 則 fallback 以 `phone_last3`（手機末 3 碼）進行比對。末 3 碼只有 1000 種組合（000–999），攻擊者即使在 per-IP rate limit（10 次/分鐘）限制下，仍可以換 IP 或分散打法在短時間內窮舉成功。

**修復**：
- 完全移除 `phone_last3` fallback，登入**只接受** `phone_hash` 精確比對。
- 移除 `MemberRow` 型別中已無用的 `phone_last3` 欄位。
- 移除 `const last3 = phone.slice(-3)` 與相關相容路徑程式碼。
- 移除成功登入後補寫 `phone_hash` 的漸進遷移邏輯（現已死碼；舊帳號須由管理員手動補齊 hash 後才能登入）。
- 保留 `phone_full` 的漸進補齊（仍有意義，與安全無關）。

**影響**：有 `phone_hash` 的帳號（含所有新帳號）不受影響。`phone_hash IS NULL` 的遺留帳號將取得 401，需管理員手動處理，詳見「部署操作」第 3 點。

---

### C-2（CRITICAL）— 缺少 CSRF 防護
**檔案**：`lib/api-helper.ts`
**問題**：
系統使用 httpOnly Cookie 驗證，所有寫入 API（POST/DELETE）未驗證請求來源。`SameSite=lax` 可阻擋跨站非同步請求，但**仍允許跨站表單 GET/POST**（top-level navigation）。若攻擊者引誘使用者開啟惡意頁面並發出表單，可以受害者身份執行打卡、修改排程、刪除標籤等操作。

**修復**：
- 在 `lib/api-helper.ts` 新增私有函式 `checkOrigin()`：
  ```ts
  async function checkOrigin(): Promise<NextResponse | null> {
    const headerStore = await headers()
    const origin = headerStore.get('origin')
    const host   = headerStore.get('host')
    if (origin && host && !origin.endsWith(host)) {
      return NextResponse.json({ ok: false, msg: 'Forbidden' }, { status: 403 })
    }
    return null
  }
  ```
- 在 `getCurrentMember()` 與 `requireAdmin()` 最前端均呼叫 `checkOrigin()`，跨站請求（Origin ≠ Host）一律回傳 **403 Forbidden**。
- 兩個函式覆蓋了**所有**需要身份驗證的寫入路由，無需修改各個 route handler。

**設計說明**：
- `headers()` 來自 `next/headers`，在 App Router Route Handler 中可直接使用，無需修改函式簽名。
- 瀏覽器發出的跨站請求會帶 `Origin`；同站請求（含 curl 無 Origin header）不受影響（`origin && host` 都真才觸發檢查）。
- 此方案為最小侵入性實作，等同 OWASP 建議的 Origin 驗證緩解策略。
